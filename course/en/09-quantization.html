Lesson 09

Quantization: shrinking the numbers without breaking the answer 📏

We convert the model's weights from 32-bit numbers to 8-bit ones: roughly four times smaller and faster on the chip — as long as we are careful about the rounding.

Where this stage stands ✅ — with a real plot twist

This stage is done, and it taught two real lessons. Lesson one: plain int8 wrecks our models' accuracy (the input signals live on very different scales), while int16×8 (8-bit weights, 16-bit activations) keeps accuracy untouched. Lesson two — bitter but important: when we actually handed the int16×8 file to the ST tool, it turned out to be unsupported, and the tool silently converts it back to float32! Details below.

The lesson: after quantization, always re-measure accuracy on the test data — and before celebrating, make sure the target toolchain actually runs that exact format.

🕵️ A detective story: the tool that silently converts your model back

ST's documentation said only float32/int8/uint8 are supported, but the docs were old, and it was worth testing the new version (4.0.1) for real. We designed an experiment whose answer was pinned down in advance — our int16×8 file is 118 KB and the float32 version of the same model is 338 KB; so after the upload, the reported weight size is the judge:

weights ≈ 118 KB → int16×8 really deployed ✓
weights ≈ 326 KB → silently converted to float32 ✗

The measured result: 326.15 KiB — exactly the float32 size. That is, the tool "accepted" the file, raised no error, but re-expanded the 8-bit weights to 32 bits and threw the int16 scheme away. (Sanity control: a genuine int8 file uploaded the same way was reported as 83.28 KiB — so the measurement method was sound.) Had we naively run a benchmark, we would have written in the paper "int16×8 ran at such-and-such speed" while in reality float32 had run — a bogus number, without ever telling a lie!

💡 The big lesson: "no error" is not the same as "it worked." Tools sometimes silently convert unsupported input into something else; a good experiment is one whose answer has two distinguishable outcomes fixed in advance. So: int16×8 is, for us, only an "offline accuracy bound" and is never reported as a deployed configuration; the paper's main deployment is float32 (the same route as the reference paper — a fair comparison).

The story 📖

Imagine you want to measure a friend's height. You have two rulers: one in millimeters with thousands of fine tick marks, and a plain one marked only in centimeters. The millimeter ruler is more precise, but honestly, for measuring height, "173 centimeters" is every bit as useful as "1730 millimeters" — and the simpler ruler is lighter and quicker to read.

Quantization is exactly this trade: instead of keeping every weight of the model as a very precise decimal number, we measure it with a coarser ruler that has only 256 tick marks. Each individual number loses a little precision, but the model's overall answer — if we do it right — stays nearly the same. In exchange the model becomes much smaller and faster; for our little chip, that is exactly what we want.

More precisely 🔬

Inside a computer, each weight is normally a decimal number of type float32 and takes 32 bits of memory. If we store the same number as an int8 (an 8-bit integer), only 8 bits are needed — so the weights shrink to roughly one quarter of their size. And there is a second gift: microcontrollers have fast circuits for integer arithmetic, so an int8 model is not just smaller but also faster to run.

One big question remains, though: an 8-bit number has only 256 possible values. How do we fit endlessly many decimal numbers into 256 slots? Answer: we divide the range of the real numbers into 256 equal steps and round each number to its nearest step:

The mapping recipe is as simple as this:
real value ≈ scale × (int8 code − zero-point)
• scale: the size of one step — the real range divided by 256.
• zero-point: the code that plays the role of zero; it need not sit in the middle, since the range of real numbers is not necessarily symmetric around zero either.

One important detail: this scale and zero-point are chosen for all the weights of a layer, not for each weight individually. So if a layer contains one outlier weight (a very large one), it stretches the range and coarsens the steps — and all the other weights, which used to sit close together, get rounded onto a handful of steps and their information is crushed. This is the main reason int8 drops accuracy on our data.

How do we know the range? calibration 🎯

To place the steps correctly, we need to know what range the numbers actually fall into. For the weights this is easy — they are right in front of us. But the intermediate outputs of the layers (activations) depend on the input. The solution: pass a few hundred real samples (our own 5-second windows) through the model and watch where each layer's numbers usually land. This is called calibration. If we pick the range wrong, either lots of steps go to waste, or the large values fall off the ruler and get clipped.

Two ways to round: PTQ and QAT ⚖️

Rounding adds a tiny error to every number. The question is when the model should get acquainted with that error:

MethodHow it worksResult
PTQ
Post-Training Quantization		 After training is finished, the weights are rounded in one go; quick and painless Sometimes costs a little accuracy
QAT
Quantization-Aware Training		 In a short retraining, rounding is simulated ("faked") so the network learns to live with it Accuracy is preserved better

With QAT, the network knows in advance that it will be measured with the coarse ruler, so it arranges its weights so the answer stays good even after rounding. In practice, our deployment files were built with PTQ (fast, with a few hundred real windows for calibration) — and since even well-made int8 drops accuracy on this data and int16×8 cannot be deployed, the paper's primary configuration stayed float32. Then we tried QAT too: a few epochs of fine-tuning with rounding simulated in the forward pass. The result? int8 accuracy rose from 86.9% to 89.8% — recovering about 3 of the 5 lost points, at the same int8 size. (The experiment ran on a width-1 2D re-expression of the model, because the QAT tooling only knows 2D layers; the computation is identical. On-device latency is not measured yet.)

In our project 🧰

• From every NAS winner model we build two TFLite files: float32 (the paper's primary configuration — the same deployment route as the reference paper) and int8 with PTQ (a smaller/faster operating point, with a measured accuracy drop).
• Interestingly, µNAS counted model size in int8 bytes from the very start (our cap: 256 KB) — that is, the search was run under the assumption of quantization.
• For calibration we used a few hundred real windows from the dataset itself.
• The TFLite files were handed to the ST Edge AI tool, and the real memory and speed were measured on the board — full numbers in the next lesson!

Good to know 💡

The record-holding paper (IEEE SPL 2026) put its model on the board without quantization, in plain float32; as a result, on the smaller board they tested (STM32F401), their Transformer model did not fit in RAM at all! The int8 stage is one of our ways to end up smaller and faster than them. For example, one of our search candidates with 72,729 parameters is about 284 KB as float — with int8 it comes down to roughly a quarter of that and stays comfortably under the 256 KB cap.
